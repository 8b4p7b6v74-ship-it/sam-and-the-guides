Navigation & Guidance of Blind and Visually-Impaired People

Camera-free swarm robotics and a cloud-connected app that give blind and visually impaired people the freedom to move independently, safely, and on their own terms — guided by two or more robots following and leading them.

Presented by Violet·Samuel·Gabriel·July 2026

Problem statement

03 · Proposed solution

Three low-cost robots that physically lead the way.

A system of three small robots plus a phone app that leads a blind traveler (Sam) along pre-mapped indoor routes. Two guide robots lead, a companion stays beside Sam, and one piece of software — the brain — coordinates all three over Bluetooth. No camera. No internet needed.

Ask & plan
  • Voice destination — "Take me to the store."
  • Route + ETA — "be there by 6:00 PM" and it departs on time by itself.
Coordinate
  • Real-time (~5 Hz) — balances speed & spacing so no one is left behind.
  • Safety — stops on obstacles, taps the arm, speaks every step; e-stop anytime.
Private & shared
  • Camera-free & offline-capable — private by design.
  • Live family view — a relative anywhere can follow the journey.
04 · Problem & solution statement

Stated plainly.

Problem statement
  • A blind or visually-impaired person cannot travel independently through an unfamiliar indoor space.
  • Every existing aid falls short: apps are audio-only — they describe, they don't lead; a guide dog is a $45–60k, multi-year wait serving <2%; a white cane misses every head-height hazard; the rest means depending on another person.
Solution statement
  • A camera-free swarm of three low-cost robots that physically lead the traveler along a mapped indoor route.
  • Two robots guide, one companion holds the group, and a cloud-connected hub coordinates them over Bluetooth with spoken, step-by-step guidance — private, offline-capable, paced by the user.
Thesis

Keep the robots simple and put the intelligence in the hub, and a coordinated swarm can deliver the one thing no audio-only aid can: physical guidance.

05 · Objectives & impact

What it must do — and why it matters.

Objectives
  • Lead a blind user from A to B along a mapped route, reliably.
  • Hold a safe formation — the slowest member (Sam) sets the pace.
  • Stop at obstacles and alert the user — light + arm tap + voice.
  • Compute ETA and support scheduled departures.
  • Stay low-cost, camera-free and offline-capable.
  • Provide live monitoring for family and caregivers.
Expected impact
  • Restores independent indoor mobility for a large, ageing, under-served population.
  • A kit could target under $2,000 — vs a $45–60k, multi-year guide dog.
  • Physically guides the user — unlike every audio-only app.
  • Fits accessibility mandates (EAA, ADA) — airports, care homes, museums, hospitals.
  • Assistive-independence programs return $4–7 per $1 (SROI) — plus dignity, employment, mental health.

Hardware
  • Each robot runs on a Raspberry Pi Pico (RP2350, MicroPython) — wheels + encoders, IMU, line sensors, HC-SR04 ultrasonic, servo "arm", LED.
  • One identical firmware on all three — roles assigned live.
Software
  • The hub — a web/phone app that plans routes, coordinates the fleet, and speaks.
  • Bluetooth to the robots · cloud sync for the live family view.
Engineered within tight constraints

A Raspberry Pi Pico is cheap and low-power — little compute, little memory. So the robots stay dumb and lean, the brain lives in the hub, and the kit targets under $2,000 — a fraction of a $45–60k guide dog.

The guarantee

10 · Core algorithm · pseudocode

The coordination loop, in plain steps.

# HUB — guide the traveler to a place
function guide(dest):
  route ← planRoute(graph, here, dest)   # BFS

  # phase 1 · preload, confirmed
  sendReliable(all, "STOP")
  for r in fleet:
    ok ← sendReliable(r, mission[r], tries=4)
    if not ok: abort("unconfirmed"); return

  # phase 2 · synchronized start
  burst(all, "GO")              # fire ×3, one tick
  wait telemetry = RUN   (≤ 1.4 s)
  if not allRunning: stopAll(); return

  # phase 3 · coordinate ~5 Hz
  while not arrived:
    hold gap(Closer) → Sam.speed    # user paces
    if front(Opener) < safe: stopAll(); alert()
    on turn: mirror → Sam, then Closer
    speak(nextStep)
  speak("You have arrived.")
# ROBOT — one shared firmware
on receive(line):
  if line starts "#id":
    emit("A|"+id)       # ACK, always
    if id in seen: return  # dedup
    seen.add(id)
  cmd ← strip(line)

  if cmd = "GO":
    if mission ≠ RUN:        # idempotent
      startMission()
  elif cmd = "STOP": halt()
  elif cmd = "PATH …": load(path)

# telemetry, ~20 Hz
every 50 ms:
  emit("T|role|mission|front|…")
Why it holds

Every command is acknowledged & retried; GO is idempotent, so a duplicate never restarts a moving robot; and the fleet is verified in telemetry before it's trusted to move.

11 · Prototype development

From three kits to a coordinated convoy.

Building process
  • Assembled 3 XRP kits; wrote one firmware shared by all — roles assigned live over Bluetooth.
  • Built the web-based Hub (Web Bluetooth) that connects the fleet, plans routes on a hand-mapped model town, speaks each step, and syncs to the cloud.
  • Added a voice interface — spoken destination in, spoken guidance out.
Initial testing
  • Verified BLE connection + role assignment and the live telemetry loop.
  • Confirmed motor control and per-robot state reporting.
  • Built a full software simulator of the 3-robot convoy to validate coordination end-to-end before touching hardware.
12 · Testing & iteration

The real challenge was wireless reliability — not the motors.

Environments
  • A hand-built miniature town, measured to the centimetre.
  • A full simulator (virtual fleet) with injectable packet loss.
Findings
  • Commands could be lost, delayed, or duplicated — a robot left behind.
  • Sensors were noisy — ultrasonic spikes, encoder glitches.
Iterations
  • Rebuilt comms: acknowledgment, retry, synchronized start, verification.
  • An adversarial review found & fixed 10 reliability defects.
  • Hardened firmware — idempotent commands, debounced obstacle detection.
✓ Delivery holds at 40% simulated packet loss → still 3/3
13 · Results & impact

A swarm you can trust to move together.

Success metrics
  • Commands confirmed by all 3 — even at 40% simulated packet loss.
  • Deterministic: the fleet starts in sync, or safely aborts.
  • Full voice → coordinate → guide → arrive validated in simulation.
User-centred design
  • Camera-free & offline-capable — answers the privacy concern.
  • The user sets the pace — the system adapts to them.
  • Spoken guidance + arm tap for a non-visual experience.
Challenges overcome
  • BLE reliability — a verifiable, retrying protocol.
  • Sensor noise — filtering & debouncing.
  • Synchronization — a two-phase, verified start.
